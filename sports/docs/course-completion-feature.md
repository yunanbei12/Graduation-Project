# 课程完成状态功能说明

## 功能概述

为私教课包增加"已完成"状态，当课包的所有课时全部消课完成后，自动将课包和对应订单状态更新为"已完成"，实现收入的准确统计。

## 业务流程

### 1. 私教课消课流程

```
用户购买课包 → 订单支付(status=2) → 订单待排课(status=3) → 课包创建(status=1)
                                                                    ↓
                                                            逐节消课(usedLessons++)
                                                                    ↓
                                                    课包全部用完(usedLessons >= totalLessons)
                                                                    ↓
                                                            课包状态=3(已完成)
                                                                    ↓
                                                            订单状态=4(已完成)
                                                            订单完成时间=当前时间
```

### 2. 标记缺勤的回滚逻辑

```
标记缺勤 → 退还课时(usedLessons--) → 检查课包是否从已完成状态恢复
                                            ↓
                                    如果 wasCompleted=true
                                            ↓
                                    课包状态=1(正常)
                                            ↓
                                    订单状态=3(待排课)
                                    订单完成时间=null
```

## 数据模型变更

### UserCoursePackage 状态枚举

| 状态值 | 状态名称 | 说明 |
|--------|---------|------|
| 0 | 已过期 | 课包超过有效期 |
| 1 | 正常 | 课包进行中，可继续消课 |
| 2 | 已退费 | 课包已退款 |
| **3** | **已完成** | **课包所有课时已消课完成（新增）** |

### Order 状态流转

**课程订单（orderType=1）**：
- 1 → 2 → 3 → **4**（新增自动流转）
- 待支付 → 已支付 → 待排课 → **已完成**

**商品订单（orderType=2）**：
- 1 → 2 → 3 → 4
- 待支付 → 已支付 → 待发货 → 已完成

   - 更新 `status` 字段注释，增加状态 3=已完成

2. **CheckinController.java**
   - `privateCheckin()` 方法：消课时判断课包是否完成，自动更新课包和订单状态
   - `markAbsent()` 方法：标记缺勤时，如果课包从已完成恢复，同步恢复订单状态

   - 将历史数据中已用完的课包状态从 0 更新为 3
   - 将已完成课包对应的订单状态更新为 4

### 前端修改

1. **sports-admin-web/src/views/finance/income.vue**

   - 支持显示 `status=3` 的"已完成"状态

## 测试验证

### 测试场景 1：新课包完成流程

1. 创建一个 12 节课的私教课包订单
2. 支付订单，验证订单状态为 3（待排课）
3. 逐节消课，验证课包 `usedLessons` 递增
4. 消完第 12 节课后，验证：
   - 课包状态 = 3（已完成）
   - 订单状态 = 4（已完成）
   - 订单 `finishTime` 已设置
5. 在财务收入页面验证该订单已被统计到收入中

### 测试场景 2：标记缺勤回滚

1. 对已完成的课包，标记最后一节课为缺勤
2. 验证：
   - 课包 `usedLessons` 减 1
   - 课包状态恢复为 1（正常）
   - 订单状态恢复为 3（待排课）
   - 订单 `finishTime` 清空
3. 在财务收入页面验证该订单不再被统计

### 测试场景 3：历史数据迁移

1. 查询迁移前已用完的课包数量
   ```sql
   SELECT COUNT(*) FROM user_course_package 
   WHERE status = 0 AND used_lessons >= total_lessons;
   ```

2. 执行迁移脚本后，验证这些课包状态已更新为 3

3. 验证对应订单状态已更新为 4

## 注意事项

1. **团课订单**：团课订单在结课时直接设为已完成（status=4），不受此次修改影响

2. **退款订单**：已退款的订单不会被统计到收入中，逻辑保持不变

3. **过期课包**：过期的课包（status=0）不会被自动更新为已完成，需要手动处理

4. **并发安全**：消课操作建议加分布式锁，避免并发消课导致数据不一致

## 收益

1. **准确的收入统计**：课程订单在课包完成后自动计入收入，无需手动干预
2. **清晰的订单状态**：订单状态流转更符合业务实际，便于运营管理
3. **完整的数据追溯**：通过订单完成时间可以准确统计各时间段的收入
4. **灵活的回滚机制**：支持标记缺勤后自动回滚状态，保证数据一致性
