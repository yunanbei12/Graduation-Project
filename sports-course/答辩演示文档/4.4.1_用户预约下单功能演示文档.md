# 4.4.1 用户预约下单功能演示文档

## 1. 功能定位

用户预约下单功能用于支撑移动端课程购买场景，用户可在小程序端完成课程浏览、场次选择、优惠券抵扣、订单提交与支付处理，形成从课程展示到交易完成的完整业务闭环。本系统同时支持私教课和团课两种预约模式：

1. 私教课采用“先购买课包，后续按次销课”的履约方式。
2. 团课采用“先选择具体场次，支付后占用名额”的履约方式。

该功能不仅要完成基本的订单创建，还需要解决重复下单、重复支付、团课超卖、待支付订单超时关闭以及团课成团判断等问题，因此是课程模块中业务规则最复杂的部分之一。

---

## 2. 演示目标

本节演示的目标主要包括以下四点：

1. 展示用户如何在小程序端完成课程预约与下单。
2. 展示系统如何在后端完成订单、优惠券和订单明细的统一保存。
3. 展示私教课与团课在支付后履约初始化逻辑上的差异。
4. 展示系统如何通过锁、事务和延迟消息保证并发场景下的数据一致性。

---

## 3. 演示准备

为保证演示过程顺利，建议提前准备以下测试数据：

1. 一个已注册、已登录且已绑定手机号的测试用户。
2. 一门可购买的私教课程，建议设置为体验课或少课时课包，便于后续演示销课。
3. 一门可购买的团课课程，并提前生成至少一个可报名场次。
4. 一张当前可用且满足使用门槛的优惠券。

---

## 4. 演示步骤

### （1）私教课下单演示

演示时可按照如下顺序进行：

1. 在小程序端进入课程列表，打开一门私教课程详情页。
2. 点击“立即购买”进入确认订单页面。
3. 选择可用优惠券，展示订单金额、优惠金额和实付金额的联动变化。
4. 提交订单并完成支付。
5. 支付成功后进入“我的订单”，展示订单状态已更新为“处理中”。
6. 进入“我的课包”，展示系统已自动生成与该订单关联的课包记录。

在这一过程中，可以重点说明：私教课支付成功后并不会占用具体场次，而是自动生成后续可销课的用户课包。

### （2）团课下单演示

演示时可继续展示团课下单链路：

1. 返回课程列表，打开一门团课课程详情页。
2. 选择具体的上课场次后进入确认订单页面。
3. 提交订单并完成支付。
4. 展示支付成功后订单状态更新为“处理中”。
5. 返回后台排课管理页，展示该场次的已报名人数已经自动增加。

在这一过程中，可以重点说明：团课订单必须绑定具体场次，且支付成功时系统才会真正占用名额。

---

## 5. 核心实现说明

### （1）订单创建与事务控制

订单创建阶段主要完成用户身份校验、课程状态校验以及订单基础信息初始化。由于订单创建同时涉及订单主表、订单项和优惠券状态等多个数据对象，系统采用事务机制统一控制处理过程，确保任一环节异常时能够整体回滚。其核心逻辑可概括如下：

```java
createCourseOrder(userId, params) {
    // 1. 校验当前用户是否存在，并确认已绑定手机号
    validateUser(userId);

    // 2. 校验课程是否存在且处于上架状态
    Course course = validateCourse(params.courseId);

    // 3. 初始化订单对象，设置订单编号和初始状态
    Order order = buildPendingOrder(userId, course, params);
}
```

### （2）团课场次校验与并发防重

当用户购买团课时，系统还需要校验所选场次是否有效，以及当前用户是否已重复报名该场次。由于团课场次属于共享资源，存在并发建单风险，因此系统在订单创建阶段对“用户 + 场次”维度加锁，避免同一用户并发重复报名。其核心逻辑如下：

```java
if (course.isGroupCourse()) {
    // 1. 校验是否选择了具体场次
    validateScheduleSelected(params.scheduleId);

    // 2. 对“用户 + 场次”加锁，避免并发重复建单
    lock(userId, scheduleId);

    try {
        // 3. 校验场次是否有效、是否可报名
        validateScheduleForSignup(course, schedule);

        // 4. 检查当前用户是否已报名该场次
        ensureNoDuplicateOrder(userId, scheduleId);

        // 5. 保存订单与场次之间的关联
        order.bindSchedule(scheduleId);
    } finally {
        unlock();
    }
}
```

### （3）优惠券计算

在订单基础信息准备完成后，系统进入金额计算阶段。该阶段需要判断用户是否选择优惠券，并校验优惠券归属、状态、有效期和使用门槛，再统一完成实付金额计算。其逻辑如下：

```java
calculateCouponPricing(userId, couponId, totalAmount) {
    // 1. 校验优惠券是否属于当前用户
    validateCouponOwner(userId, couponId);

    // 2. 校验优惠券状态、有效期和使用门槛
    validateCouponAvailable(couponId, totalAmount);

    // 3. 根据优惠券类型计算优惠金额
    return couponAmount;
}
```

### （4）订单保存与延迟关单

完成课程校验、场次校验和优惠计算后，系统正式生成订单主表和订单项记录，并在事务提交成功后发送延迟关单消息，用于处理待支付订单超时自动取消的问题。其核心逻辑如下：

```java
saveOrder(order, course, pricing) {
    // 1. 保存订单主表
    persistOrder(order, pricing);

    // 2. 保存订单明细
    persistOrderItem(order, course);

    // 3. 锁定优惠券
    lockCouponForOrder(order);

    // 4. 在事务提交成功后发送延迟关单消息
    afterCommit(() -> publishOrderCloseDelay(order.id));
}
```

### （5）订单支付与并发控制

支付阶段的重点在于防止同一订单被重复支付，并在团课场景下避免场次超卖。系统在支付环节使用订单锁和场次锁两级并发控制机制，支付成功后还会触发订单支付事件和团课成团检查调度。其核心逻辑如下：

```java
payOrder(userId, orderId) {
    // 1. 校验订单归属
    Order order = requireOwnedOrder(userId, orderId);

    // 2. 对订单加锁，防止重复支付
    lock(orderId);

    try {
        // 3. 重新读取订单，只允许待支付订单继续处理
        Order latest = validatePendingOrder(orderId);

        // 4. 根据课程类型执行支付后的履约初始化
        handleCoursePay(latest);

        // 5. 更新支付时间并核销已锁定优惠券
        markPaid(latest);
        consumeLockedCoupon(latest);

        // 6. 团课支付成功后触发成团检查调度
        if (latest.hasSchedule()) {
            publishGroupCheckEventIfAbsent(latest.scheduleId);
        }

        // 7. 在事务提交成功后发布支付完成事件
        afterCommit(() -> publishOrderPaid(latest));
    } finally {
        unlock();
    }
}
```

### （6）支付后的履约初始化

支付成功后，系统需要根据课程类型分别执行不同的履约初始化逻辑。私教课支付后直接生成用户课包；团课支付后则占用具体场次名额，并在支付阶段再次校验场次状态，以防止超卖。其逻辑如下：

```java
handleCoursePay(order) {
    Course course = getCourse(order.courseId);

    if (course.isPrivateCourse()) {
        // 私教课：支付后生成用户课包
        createUserCoursePackage(order, course);
        order.status = PROCESSING;
        return;
    }

    // 团课：支付后占用场次名额
    lock(scheduleId);
    try {
        validateScheduleForSignup(course, schedule);
        increaseEnrolledSeats(schedule);
    } finally {
        unlock();
    }

    order.status = PROCESSING;
}
```

### （7）团课延迟成团判断

团课场次的最终状态不能在下单时立即确定，而是需要在临近开课时根据实际报名人数判断是否成团。系统在团课支付成功后，通过延迟消息机制异步调度成团检查任务，并利用Redis标记避免同一场次重复投递消息。其逻辑如下：

```java
publishGroupCheckEventIfAbsent(scheduleId) {
    // 1. 计算成团检查时刻
    checkTime = resolveCheckTime(scheduleId);

    // 2. 使用Redis标记控制同一场次只调度一次
    if (firstSchedule(scheduleId)) {
        // 3. 投递延迟消息，等待到达检查时刻后执行成团判断
        publishGroupCheckDelay(scheduleId, checkTime);
    }
}
```

---

## 6. 演示讲解重点

正式展示时，建议重点强调以下内容：

1. 私教课和团课虽然都走订单链路，但履约初始化逻辑完全不同。
2. 团课订单在创建阶段防重复，在支付阶段防超卖，体现了并发控制设计。
3. 优惠券不是前端简单计算，而是由后端统一校验和计算，保证金额口径一致。
4. 延迟关单和成团检查都通过事务提交后事件和延迟消息机制实现，体现了系统的工程化设计。

---

## 7. 总结

用户预约下单功能并不是单一的订单保存过程，而是一个融合订单创建、优惠计算、并发控制、支付处理和履约初始化的完整交易链路。该功能既支撑了课程模块的核心交易能力，也为后续的排课、销课、签到、退款和AI咨询提供了真实、连续的业务数据基础。
