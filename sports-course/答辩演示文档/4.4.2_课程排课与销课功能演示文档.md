# 4.4.2 课程排课与销课功能演示文档

## 1. 功能定位

课程排课与销课功能是体育教培机构管理系统的重要履约环节，主要负责团课排期维护、私教销课、团课结课签到以及订单履约状态联动。该功能既决定了课程能否正常交付，也直接影响订单状态、课包状态和考勤记录的准确性，因此是课程模块中连接“交易完成”与“实际履约”的关键部分。

---

## 2. 演示目标

本节演示的目标主要包括以下四点：

1. 展示后台如何为团课创建排课场次。
2. 展示系统如何检测教练时间冲突与重复排课问题。
3. 展示私教销课后课包状态与订单状态的联动更新。
4. 展示团课结课时如何批量生成签到记录并收口场次状态。

---

## 3. 演示准备

为保证演示过程顺利，建议提前准备以下测试数据：

1. 一门已启用的团课课程，并配置好开始时间、结束时间和成团人数。
2. 一名状态正常的教练。
3. 一门已购买的私教课程课包，建议剩余课时较少，便于现场演示“销课后完成”。
4. 一条已报名成功的团课场次记录，便于演示批量结课和签到。

---

## 4. 演示步骤

### （1）团课排课演示

演示时可按如下顺序进行：

1. 在后台进入“排课管理”页面。
2. 新建一条团课排课记录，选择课程、教练、日期和总座位数。
3. 展示系统成功保存排课结果。
4. 如时间允许，可尝试用同一教练、同一时间段再创建一条排课，展示系统给出冲突提示。

该过程可重点说明：系统在排课阶段已经提前处理了重复排课和教练时间冲突问题。

### （2）私教销课演示

演示时可按如下顺序进行：

1. 在后台进入“销课管理”页面。
2. 选择一名已有私教课包的学员。
3. 选择对应课程、教练和上课时间，执行一次私教销课。
4. 展示系统生成销课记录，并同步扣减课包剩余课时。
5. 若该课包课时已全部用完，则展示对应订单状态同步更新为“已完成”。

### （3）团课结课演示

演示时可继续展示团课结课链路：

1. 在后台选择一个已报名且课程时间已结束的团课场次。
2. 执行“批量结课”操作。
3. 对正常出勤学员与缺席学员分别展示不同签到状态。
4. 展示场次下相关订单状态统一更新为“已完成”。
5. 展示排课状态同步更新为“已结课”。

---

## 5. 核心实现说明

### （1）排课创建与教练冲突检测

排课创建阶段需要将课程、教练和时间安排组织为具体场次记录。由于同一教练在同一时间段只能教授一门课程，系统在保存团课排课时依次完成必填项校验、课程校验、教练校验、排课约束校验、重复排课校验和教练时间冲突检测，并通过事务机制保证排课保存过程的原子性。其核心逻辑如下：

```java
saveGroupSchedule(schedule) {
    // 1. 校验课程、教练、日期和总座位数等基础信息
    validateRequiredFields(schedule);

    // 2. 校验课程和教练状态是否有效
    validateCourse(schedule.courseId);
    validateCoach(schedule.coachId);

    // 3. 校验排课日期、成团人数和重复排课
    validateScheduleConstraints(schedule);
    validateScheduleDuplicate(schedule);

    // 4. 规范化排课时间和地点信息
    normalizeSchedule(schedule);

    // 5. 检测教练时间冲突
    validateCoachConflict(schedule);

    // 6. 保存排课
    save(schedule);
}
```

其中，`validateCoachConflict()` 方法会将目标排课与该教练已有排课的时间区间进行比较，若存在重叠，则直接拒绝保存，从而避免同一教练在同一时间段承担多门课程。

### （2）私教销课与状态联动

私教销课需要处理销课记录、课包状态和订单状态三者之间的联动关系。后台在私教销课时，首先定位当前学员可用的课包，然后生成销课记录，并同步更新课包已用课时；当课包课时全部消耗完毕时，系统自动将课包状态更新为已完成，并将对应订单状态同步更新为已完成，从而保证订单生命周期与实际履约进度保持一致。其处理逻辑如下：

```java
privateCheckin(request) {
    // 1. 生成私教销课记录
    CourseCheckin checkin = buildPrivateCheckin(pkg, request);
    saveCheckin(checkin);

    // 2. 更新课包已用课时
    pkg.usedLessons = pkg.usedLessons + 1;

    // 3. 如果课时已全部消耗，则将课包状态更新为已完成
    if (pkg.usedLessons >= pkg.totalLessons) {
        pkg.status = COMPLETED;
    }
    updatePackage(pkg);

    // 4. 若课包已完成，且存在关联订单，则同步更新订单状态
    if (pkg.isCompleted() && pkg.hasOrder()) {
        Order order = getOrder(pkg.orderId);
        if (order != null && order.isProcessing()) {
            order.status = FINISHED;
            order.finishTime = now();
            updateOrder(order);
        }
    }
}
```

### （3）团课批量签到与结课

团课结课时需要为所有报名学员生成签到记录，并同步收口订单与排课状态。系统首先校验课程是否已真正结束，然后查询该场次下已生效的报名订单，并为每位学员批量构建签到记录；缺席学员会被标记为未出勤，其余学员则记录为正常出勤。完成签到保存后，系统将相关订单状态统一更新为已完成，并将排课状态更新为已结课。其处理逻辑如下：

```java
settleGroupSchedule(scheduleId, absentUserIds) {
    // 1. 校验当前课程是否已结束，未结束则禁止结课
    validateScheduleFinished(scheduleId);

    // 2. 查询该场次下已生效的报名订单
    List<Order> paidOrders = listEffectiveOrders(scheduleId);

    // 3. 遍历订单，为每位学员生成团课签到记录
    for (Order order : paidOrders) {
        CourseCheckin checkin = new CourseCheckin();
        checkin.userId = order.userId;
        checkin.scheduleId = scheduleId;
        checkin.checkinType = GROUP_CHECKIN;

        // 缺席学员记为未出勤，其余学员记为正常出勤
        if (absentUserIds.contains(order.userId)) {
            checkin.status = ABSENT;
        } else {
            checkin.status = ATTENDED;
        }

        addToCheckinList(checkin);

        // 4. 同步更新该订单状态为已完成
        if (order.isNotFinished()) {
            order.status = FINISHED;
            order.finishTime = now();
            updateOrder(order);
        }
    }

    // 5. 批量保存签到记录
    saveBatch(checkins);

    // 6. 全部处理完成后，将排课状态更新为已结课
    schedule.status = SCHEDULE_FINISHED;
    updateSchedule(schedule);
}
```

### （4）定时任务兜底机制

对于未成团的场次，系统通过定时任务自动处理：每30分钟扫描一次已到开课时间但仍未成团的团课场次，并调用统一的成团判断逻辑执行取消和退款操作；对于已经结束但尚未人工结课的团课，系统不会自动生成签到记录，而是仅记录日志提醒管理员及时处理，以保证考勤数据的准确性。其核心逻辑如下：

```java
updateExpiredScheduleStatus() {
    // 1. 处理未成团的过期团课
    cancelUngroupedSchedules();

    // 2. 检查已结束但未人工结课的团课
    finishCompletedSchedules();
}
```

在该机制中，系统通过“自动取消 + 人工确认结课”的处理方式，在提升状态收口效率的同时，保证了考勤数据的可靠性。

---

## 6. 演示讲解重点

正式展示时，建议重点强调以下内容：

1. 排课不是简单保存一条记录，而是要同时校验课程、教练、时间和成团人数。
2. 私教销课的重点不是单次打卡，而是课包、销课记录和订单状态的联动。
3. 团课结课采用批量处理方式，可以一次性完成签到、订单收口和排课收口。
4. 定时任务机制保证了未成团场次和已结束场次即使没有人工及时操作，也能够得到合理处理或提醒。

---

## 7. 总结

课程排课与销课功能并不是独立存在的后台操作模块，而是课程交易闭环中的核心履约环节。该模块通过排课校验、销课联动、批量结课和定时任务兜底机制，实现了从课程安排到履约完成的全过程管理，也为订单状态、课包状态和考勤数据提供了统一且可靠的业务支撑。
