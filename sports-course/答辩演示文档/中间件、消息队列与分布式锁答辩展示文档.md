# 中间件、消息队列与分布式锁答辩展示文档

## 1. 功能定位

本系统在课程预约下单、订单支付、团课成团判断和自动退款等关键业务环节中，引入了 Redis 分布式锁、RabbitMQ 延迟消息以及事务提交后事件发布机制，用于解决并发下单、重复支付、场次超卖、待支付订单超时滞留以及异步状态收口等问题。相较于单纯依赖数据库事务的实现方式，该方案能够在分布式部署和高并发业务场景下更稳定地保证订单数据、排课数据和优惠券状态的一致性。

---

## 2. 展示目标

本节展示的目标主要包括以下五点：

1. 展示系统为何在订单模块中引入 Redis 分布式锁。
2. 展示 RabbitMQ 在延迟关单、成团检查和订单事件通知中的具体作用。
3. 展示事务提交后再发送消息的实现思路及其必要性。
4. 展示系统如何通过状态校验、锁控制和消息消费机制实现幂等处理。
5. 展示系统在消费异常、重复投递和业务状态变化场景下的兜底策略。

---

## 3. 中间件总体链路

本系统订单链路中的中间件协作过程可概括为以下步骤：

1. 用户提交订单后，系统在事务内完成订单、订单项和优惠券锁定等数据写入。
2. 本地事务提交成功后，再通过 `afterCommit()` 发布延迟关单消息，防止未支付订单长期占用资源。
3. 用户发起支付后，系统先对订单加锁，防止同一订单被重复支付。
4. 若为团课订单，系统在支付阶段继续对场次加锁，并再次校验名额，防止支付时超卖。
5. 团课支付成功后，系统仅允许首个成功支付的订单投递成团检查延迟消息，避免重复调度。
6. 延迟消息到达后，消费者异步执行超时关单或成团判断逻辑，并根据结果自动取消订单或退款。

从实现方式来看，本系统并不是简单地“加一个队列”或“加一个锁”，而是将事务、锁和消息机制组合使用，形成了较完整的一致性控制链路。

## 4. 核心实现说明

### （1）Redis 分布式锁设计

本系统在订单模块中一共使用了四类 Redis 分布式锁，不同锁负责不同粒度的并发控制：

1. `order:user:schedule:create:userId:scheduleId`
   用于团课下单阶段，控制“同一用户对同一场次”的并发建单，防止重复报名。
2. `order:pay:orderId`
   用于支付阶段，保证同一订单在任意时刻只能有一个支付流程进入执行。
3. `order:schedule:scheduleId`
   用于团课支付和退款阶段，保证场次已报名人数的增减操作串行化，防止超卖或人数回滚错误。
4. `order:schedule:check:scheduleId`
   用于成团检查阶段，避免同一场次被多个消费者或定时任务重复执行成团判断。

该设计说明系统并未采用“一把大锁”处理所有订单，而是按照“用户-场次”“订单”“场次”“场次检查”四个粒度分别加锁，从而在保证一致性的同时尽量降低锁竞争范围。

其核心逻辑如下：

```java
createOrPayOrder() {
    // 团课建单时：锁住“用户 + 场次”，防止并发重复报名
    lock("order:user:schedule:create:userId:scheduleId");

    // 支付时：锁住订单，防止同一订单重复支付
    lock("order:pay:orderId");

    // 团课支付或退款时：锁住场次，串行处理名额增减
    lock("order:schedule:scheduleId");

    // 成团检查时：锁住场次检查任务，避免重复执行
    lock("order:schedule:check:scheduleId");
}
```

这里选择 Redis 分布式锁，而不是 `synchronized` 或单纯数据库锁，主要原因在于系统服务可分布式部署，而本地 JVM 锁无法跨进程生效；数据库锁虽然也能控制并发，但更适合短事务内的行级竞争，不适合覆盖支付、退款和异步检查这类跨服务、跨流程的业务控制场景。

### （2）事务提交后事件发布机制

为避免“数据库事务尚未提交，但消息已经先发出”的问题，系统在订单创建、支付完成和退款完成等场景下，均通过 `TxEventHelper.afterCommit()` 在事务真正提交成功后再发布 MQ 事件。这样可以保证消费者读取到的订单状态一定是已落库的最终结果。

其核心逻辑如下：

```java
saveOrder() {
    // 1. 事务内保存订单、订单项和优惠券状态
    persistOrderData();

    // 2. 事务提交成功后再发送 MQ 消息
    afterCommit(() -> publishOrderCloseDelay(orderId));
}
```

该机制的技术意义在于：它解决的是“消息发送时机”和“本地事务提交时机”之间的一致性问题。若不加这一层控制，可能出现消息已被消费者处理，但订单数据最终回滚的异常情况，从而导致关单、成团判断或活动发券逻辑基于错误数据执行。

### （3）RabbitMQ 延迟消息设计

本系统在订单模块中主要使用了两类延迟消息：

1. 待支付订单超时关闭消息。
2. 团课场次成团检查消息。

两类消息虽然都属于“延迟执行”，但实现方式并不完全相同：

1. 延迟关单采用“队列 TTL + 死信转发”方式实现，延迟时间由 `close-timeout-minutes` 统一配置，当前默认值为 15 分钟。
2. 成团检查采用“消息级别 expiration + 死信转发”方式实现，延迟时间根据每个场次的开课时间动态计算，更适合处理不同场次的差异化调度需求。

其核心逻辑如下：

```java
publishDelayEvent() {
    // 延迟关单：固定超时时间，适合使用队列 TTL
    publishToOrderCloseDelayQueue(orderId);

    // 成团检查：每个场次开课时间不同，适合使用消息级延迟
    publishToGroupCheckDelayQueue(scheduleId, delayMillis);
}
```

从实际配置上看，系统定义了 `kinetic.order.event.exchange` 和 `kinetic.group.event.exchange` 两个交换机，并分别对应订单关闭、成团检查、支付完成和退款完成等不同事件队列，从而实现了订单事件与团课事件的分离处理。

### （4）支付后的异步事件链路

当用户完成支付后，系统并不会在一个方法中同步处理全部后续动作，而是将“核心支付动作”和“后续扩展动作”进行拆分：

1. 同步阶段只负责支付校验、状态更新、优惠券核销和团课名额占用。
2. 异步阶段通过 MQ 继续处理订单支付成功事件，例如活动优惠券发放、团课成团检查调度等操作。

其核心逻辑如下：

```java
payOrder(orderId) {
    // 1. 对订单加锁，防止重复支付
    lock("order:pay:orderId");

    // 2. 只允许待支付订单进入支付流程
    validatePendingOrder(orderId);

    // 3. 根据课程类型执行履约初始化
    handleCoursePay(order);

    // 4. 更新支付时间并核销优惠券
    markPaid(order);
    consumeLockedCoupon(order);

    // 5. 团课支付成功后，尝试调度成团检查
    if (order.hasSchedule()) {
        publishGroupCheckEventIfAbsent(order.scheduleId);
    }

    // 6. 事务提交成功后，再发布支付完成事件
    afterCommit(() -> publishOrderPaid(order));
}
```

该实现体现出两个关键点：一是支付逻辑必须串行化；二是支付成功后的非核心动作应尽量异步化，避免在支付接口中堆积过多后置业务，影响接口响应时间和稳定性。

### （5）团课成团检查与防重复调度

团课支付成功后，系统会根据“开课时间 - 提前判断分钟数”计算成团检查时间，并尝试向 Redis 写入一个带过期时间的标记：

1. 若当前场次尚未被调度过，则写入成功，并发送成团检查延迟消息。
2. 若标记已存在，则说明此前已有订单为该场次完成了调度，本次不再重复投递。

其核心逻辑如下：

```java
publishGroupCheckEventIfAbsent(scheduleId) {
    // 1. 根据开课时间计算成团检查时刻
    checkTime = scheduleStart - groupCheckBeforeMinutes;

    // 2. 计算消息延迟时间和 Redis 标记过期时间
    delayMillis = between(now, checkTime);
    ttlSeconds = between(now, scheduleStart);

    // 3. 仅允许首个支付成功的订单完成调度
    if (trySetRedisFlag(scheduleId, ttlSeconds)) {
        publishGroupCheckDelay(scheduleId, delayMillis);
    }
}
```

这种“Redis 标记 + 延迟消息”的组合设计，能够有效避免一个团课场次在多次支付成功后重复投递多条成团检查消息。

### （6）消息消费、异常处理与重试策略

在消费端，系统对不同类型异常采取了区分处理策略：

1. 若属于业务可接受异常，例如订单已支付、订单已关闭、场次已取消等情况，则记录日志后直接跳过，不再重试。
2. 若属于系统异常，例如数据库访问失败或消费逻辑执行异常，则抛出异常，由消息中间件重新投递。

当前配置中，API 服务启用了 `default-requeue-rejected: true`，因此当监听器抛出异常时，消息会重新入队等待后续消费。其核心逻辑如下：

```java
consumeDelayEvent(message) {
    try {
        // 正常执行业务逻辑，如自动关单或成团检查
        handleBusiness(message);
    } catch (IllegalArgumentException e) {
        // 业务状态已变化，记录日志后跳过
        logAndSkip(e);
    } catch (Exception e) {
        // 系统异常重新抛出，交由 MQ 重试
        throw e;
    }
}
```

这种处理方式避免了无意义的重复消费，也提高了系统面对瞬时故障时的恢复能力。

### （7）幂等性与一致性保障

本系统并未单独设计一张“幂等表”，而是采用“状态机校验 + 分布式锁 + 条件更新 + 事件去重标记”的组合方式实现业务幂等，主要体现在以下几个方面：

1. 订单支付前必须再次校验订单状态是否仍为待支付。
2. 团课支付时必须再次校验场次是否仍允许报名，防止下单通过但支付时名额已满。
3. 成团检查只允许同一场次在同一时刻由一个线程执行。
4. 团课成团检查消息只允许同一场次投递一次。
5. 延迟关单、自动退款等消费者即使收到重复消息，也会先基于当前业务状态进行判断，不满足条件则直接跳过。

因此，系统的一致性保障并不是依赖某一个单点机制，而是依赖多层校验共同完成。

## 5. 配置要点

结合当前项目实现，答辩时可重点说明以下配置口径：

1. `biz.order.rabbit-enabled=true` 时，API 服务启用订单事件链路。
2. 当前开发配置中，管理端 `biz.order.rabbit-enabled=false`，因此不会注册订单事件监听器。
3. `close-timeout-minutes=15` 表示待支付订单默认 15 分钟自动关闭。
4. `group-check-before-minutes=0` 表示团课默认在开课时执行成团判断。

这说明系统并不是无差别地在所有服务中启用 MQ 消费，而是通过配置控制实际生效的事件处理节点。

## 6. 演示讲解重点

正式展示时，建议重点强调以下内容：

1. Redis 锁的重点不在“加了锁”，而在于锁粒度划分是否合理，本系统按照不同业务对象拆分了四类锁。
2. RabbitMQ 的重点不在“发了消息”，而在于延迟消息是否真正参与了订单超时关闭和团课异步成团判断。
3. `afterCommit()` 的重点不在于封装了一个工具类，而在于保证消息发送发生在事务提交成功之后。
4. 幂等控制的重点不在于是否有独立幂等表，而在于系统是否能在重复请求、重复消费和状态变化场景下稳定收口。
5. 团课支付阶段再次校验场次名额，是防止超卖的关键步骤，也是最容易被老师追问的地方之一。

---

## 7. 总结

本系统在订单与履约链路中，通过 Redis 分布式锁解决并发控制问题，通过 RabbitMQ 延迟消息解决异步调度和状态收口问题，并通过事务提交后事件发布机制保证本地事务与消息发送时机的一致性。三者配合后，系统能够较稳定地处理重复下单、重复支付、团课超卖、订单超时未支付和团课未成团自动取消等复杂业务场景，也使整个课程交易与履约流程具备了更强的工程化和可扩展能力。
