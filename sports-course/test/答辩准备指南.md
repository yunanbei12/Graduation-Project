# 体育教培机构管理系统 —— 答辩准备指南

## 一、项目全貌

### 1.1 系统简介

本系统是一个面向体育教培机构的综合管理平台，支持**私教课**和**团课**两种教学模式，覆盖课程管理、订单支付、排课销课、教练结算、AI 客服、个性化推荐等完整业务流程。

## 二、三层架构与请求追踪

### 2.1 三层架构

以"学员购买私教课"为例，一个请求经过三层：

```
小程序发送: POST /order/course  { courseId: 5, couponId: 12 }
       │
       ▼
┌──────────────────────────────────────────────────┐
│  Controller 层（接待员）                           │
│  ApiOrderController.createCourseOrder()           │
│  - 接收 HTTP 请求                                  │
│  - 从 token 中解析出当前用户 ID                      │
│  - 调用 Service 层                                 │
│  - 返回统一响应格式 ServerResponseEntity            │
└──────────────────────────────────────────────────┘
       │
       ▼
┌──────────────────────────────────────────────────┐
│  Service 层（业务逻辑）                             │
│  OrderServiceImpl.createCourseOrder()             │
│  - 校验用户存在、手机已绑定                           │
│  - 校验课程存在且在售                                │
│  - 校验团课排课信息（名额、时间、重复报名）              │
│  - 计算优惠券抵扣金额                                │
│  - 生成订单号、保存订单、创建订单项                     │
│  - 锁定优惠券                                       │
│  - 发送延迟取消消息（15 分钟未支付自动取消）             │
└──────────────────────────────────────────────────┘
       │
       ▼
┌──────────────────────────────────────────────────┐
│  Mapper 层（数据库操作）                            │
│  OrderMapper.insert(order)                        │
│  - MyBatis-Plus 自动生成 SQL                      │
│  - 将 Java 对象映射到数据表                          │
└──────────────────────────────────────────────────┘
       │
       ▼
  MySQL 数据库 (kinetic_sports_course)
```

### 2.2 关键代码对照

**Controller 层** (`ApiOrderController.java`):

```java
@RestController
@RequestMapping("/order")
@RequiredArgsConstructor
public class ApiOrderController {

    private final OrderService orderService;  // Spring 自动注入

    @PostMapping("/course")
    public ServerResponseEntity<Order> createCourseOrder(@RequestBody Order order) {
        Long userId = StpUtil.getLoginIdAsLong();  // 从 token 解析当前用户
        return ServerResponseEntity.success(
            orderService.createCourseOrder(userId, order)
        );
    }
}
```

- `@RestController`：标记这是一个处理 HTTP 请求的类
- `@PostMapping("/course")`：接收 POST 请求，完整地址为 `/order/course`
- `@RequestBody`：把前端传来的 JSON 自动转换为 Java 对象
- `ServerResponseEntity.success()`：统一返回 `{"code": 200, "msg": "success", "data": ...}`

**Service 层** (`OrderServiceImpl.java` 核心逻辑):

```java
@Service
@RequiredArgsConstructor
public class OrderServiceImpl extends ServiceImpl<OrderMapper, Order>
        implements OrderService {

    @Override
    @Transactional(rollbackFor = Exception.class)  // 事务：出错自动回滚
    public Order createCourseOrder(Long userId, Order params) {
        // 1. 校验用户
        User currentUser = requireUser(userId);
        ensurePhoneBound(currentUser);

        // 2. 校验课程
        Course course = courseService.getById(params.getCourseId());
        if (course == null || course.getStatus() != 1) {
            throw new IllegalArgumentException("课程不存在或已下架");
        }

        // 3. 团课特殊处理：分布式锁防止并发重复报名
        if (Objects.equals(course.getType(), 2)) {
            RLock lock = redissonClient.getLock(
                "order:user:schedule:create:" + userId + ":" + params.getScheduleId()
            );
            lock.lock();  // 加锁
            try {
                // 校验排课状态、名额、是否已报名
                validateScheduleForSignup(course, schedule);
            } finally {
                lock.unlock();
            }
        }

        // 4. 计算价格和优惠券
        BigDecimal totalAmount = course.getPrice();
        CouponPricing pricing = calculateCouponPricing(userId, params.getCouponId(), totalAmount);
        order.setTotalAmount(totalAmount);
        order.setCouponAmount(pricing.couponAmount());
        order.setActualAmount(totalAmount.subtract(pricing.couponAmount()));
        //                  实付金额 = 原价 - 优惠金额

        // 5. 保存订单
        order.setOrderNumber(generateOrderNumber());  // 格式: KS + 时间戳 + 6位随机数
        order.setStatus(1);  // 1 = 待支付
        this.save(order);

        // 6. 创建订单项
        OrderItem item = new OrderItem();
        item.setOrderId(order.getId());
        item.setItemId(course.getId());
        item.setItemName(course.getName());
        item.setPrice(course.getPrice());
        orderItemService.save(item);

        // 7. 锁定优惠券 + 发送延迟取消消息
        lockCouponForOrder(pricing.userCouponId(), userId, order.getId());
        txEventHelper.afterCommit(() ->
            orderEventPublisher.publishOrderCloseDelay(order.getId())
        );

        return order;
    }
}
```

- `@Service`：标记业务层类，交给 Spring 管理
- `@Transactional`：声明事务，方法内任何一步出错，数据库操作全部回滚
- `extends ServiceImpl<OrderMapper, Order>`：继承后自带 `save()`、`getById()`、`updateById()` 等方法
- `redissonClient.getLock()`：分布式锁，防止并发问题

**Mapper 层** (`OrderMapper.java`):

```java
@Mapper
public interface OrderMapper extends BaseMapper<Order> {
}
```

仅 3 行代码！`BaseMapper<Order>` 自带了增删改查所有基础方法，MyBatis-Plus 自动生成 SQL。

### 2.3 统一响应格式

```java
@Data
public class ServerResponseEntity<T> {

### 3.3 订单状态机

```

## 四、关键技术解释

### 4.1 Spring Boot

**核心概念：IoC（控制反转）和 DI（依赖注入）**

传统方式（Java SE）：
```java
UserService userService = new UserService();  // 手动 new
UserDao userDao = new UserDao();              // 手动 new
userService.setUserDao(userDao);              // 手动组装
```

Spring Boot 方式：
```java
@Service
public class UserServiceImpl {
    private final UserMapper userMapper;  // Spring 自动注入，不需要 new
}
```

所有带 `@Service`、`@RestController`、`@Mapper` 等注解的类，都由 Spring 容器统一创建和管理。`@RequiredArgsConstructor` 是 Lombok 注解，自动生成带 final 字段的构造方法，Spring 通过该构造方法注入依赖。

### 4.2 MyBatis-Plus

**核心概念：ORM（对象关系映射）—— 用操作 Java 对象的方式操作数据库**

传统 JDBC 方式需要手写 SQL、手动处理 ResultSet：
```java
Connection conn = DriverManager.getConnection(...);
PreparedStatement ps = conn.prepareStatement("SELECT * FROM course WHERE id = ?");
ps.setLong(1, courseId);
ResultSet rs = ps.executeQuery();
while (rs.next()) {
    Course c = new Course();
    c.setId(rs.getLong("id"));
    c.setName(rs.getString("name"));
    // ... 每个字段都要手动 set
}
```

MyBatis-Plus 方式——只需继承 `BaseMapper`：
```java
@Mapper
public interface CourseMapper extends BaseMapper<Course> {
}
```

然后直接调用：
```java
Course course = courseMapper.selectById(courseId);                    // 按 ID 查
List<Course> list = courseMapper.selectList(wrapper);                 // 条件查询
Page<Course> page = courseMapper.selectPage(page, wrapper);           // 分页查询
courseMapper.insert(course);                                          // 插入
courseMapper.updateById(course);                                      // 更新
```

复杂条件用 `LambdaQueryWrapper`：
```java
LambdaQueryWrapper<Order> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(Order::getUserId, userId)        // WHERE user_id = ?
       .eq(Order::getStatus, 1)              // AND status = 1
       .orderByDesc(Order::getCreateTime);   // ORDER BY create_time DESC
List<Order> orders = orderMapper.selectList(wrapper);
```

### 4.3 Sa-Token（认证鉴权）

替代传统 Java Web 中的 Session 机制，使用 Redis 存储登录状态，支持分布式部署。

核心方法：
```java
StpUtil.login(userId);              // 登录，生成 token
Long userId = StpUtil.getLoginIdAsLong();  // 从请求中解析当前用户 ID
StpUtil.isLogin();                  // 判断是否登录
StpUtil.hasPermission("xxx");       // 判断是否有某权限
StpUtil.hasRole("admin");           // 判断是否有某角色
```

### 4.4 Redisson（分布式锁）

当多个服务器实例同时处理请求时，用 Redis 实现跨 JVM 的锁机制。

```java
RLock lock = redissonClient.getLock("order:pay:" + orderId);
lock.lock();
try {
    // 业务逻辑 —— 同一时刻只有一个线程能执行
} finally {
    lock.unlock();
}
```

### 4.5 RabbitMQ（消息队列）

用于异步解耦和延迟任务。项目中三个核心使用场景：

1. **订单超时取消**：下单时发送一条 15 分钟后生效的延迟消息，到时检查订单是否仍未支付，是则自动取消
2. **团课成团判断**：支付后发布延迟消息，在开课前指定时间检查是否达到最低成团人数
3. **订单生命周期通知**：支付、退款等关键事件发生时，发布消息通知其他模块

### 4.6 Lombok

编译时自动生成样板代码的注解处理器：

| 注解 | 作用 |
|------|------|
| `@Data` | 生成 getter/setter/toString/equals/hashCode |
| `@RequiredArgsConstructor` | 为 final 字段生成构造方法 |
| `@Slf4j` | 生成 log 对象 |

## 五、创新亮点

### 5.1 AI 智能客服

- **知识库管理**：管理员可在后台维护机构相关知识（课程信息、优惠政策、上课须知等）
- **智能对话**：接入大语言模型 API，学员在小程序 AI 客服页面提问，系统基于知识库内容进行回答
- **人工接管**：AI 无法解答时自动转人工客服
- **反馈统计**：记录用户反馈，分析客服质量

相关表：`ai_session`（会话）、`ai_message`（消息）、`ai_knowledge`（知识库）、`ai_feedback`（反馈）、`ai_handover`（转人工记录）

### 5.2 个性化推荐系统

基于用户行为数据进行课程推荐：
- 记录用户浏览、搜索、下单等行为到 `user_behavior` 表
- 基于协同过滤算法计算推荐结果
- 在首页展示个性化推荐课程

相关表：`user_behavior`（用户行为）

### 5.3 订单延迟消息机制

通过 RabbitMQ 的延迟队列实现：
- **下单后超时取消**：15 分钟内未支付自动取消并释放优惠券
- **团课成团判断**：在开课前特定时间自动判断是否成团，未成团则取消排课并自动退款
- **事务提交后发消息**：通过 `TxEventHelper.afterCommit()` 确保数据库事务提交后消息才发出

### 5.4 团课成团机制

团课有最低成团人数（`course.min_group_size`），流程如下：

1. 用户报名团课时，排课状态为"未开始"
2. 每次支付成功后，检查已报名人数是否达到最低成团人数
3. 达到则排课状态变为"进行中"
4. 在开课前指定时间（`OrderBizProperties.groupCheckBeforeMinutes`）做最终判断：
   - 已达到最低人数 → 成团成功
   - 未达到 → 排课取消，所有已支付订单自动退款，待支付订单自动取消

## 六、答辩常见问题准备

### Q1：介绍一下你的系统

**回答思路**：先一句话概括，再说面向的用户、核心功能、技术栈。

> "我做的是一套面向体育教培机构的综合管理系统。系统有三个端：学员使用的微信小程序、机构管理员使用的 Web 管理后台，以及后端 Java 服务。核心功能包括课程管理、订单支付、排课销课、教练结算，以及 AI 智能客服和个性化推荐两个创新功能。技术栈采用 Spring Boot + MyBatis-Plus 后端，Vue 3 管理后台，uni-app 小程序。"

### Q2：为什么选择这些技术？

**回答思路**：

> "Spring Boot 是目前 Java 生态最主流的企业级开发框架，生态成熟、社区活跃。MyBatis-Plus 在 MyBatis 基础上提供了更便捷的 CRUD 操作，可以减少大量重复的 SQL 编写工作。Sa-Token 比 Spring Security 更轻量，配置简单，适合中小型项目。Vue 3 相比 Vue 2 性能更好，Composition API 使代码组织更清晰。uni-app 一套代码可以编译到多个小程序平台，降低多端开发成本。"

### Q3：数据库是怎么设计的？有多少张表？

**回答思路**：先说核心实体，再说关系。

> "数据库共 27 张表，按业务模块分为：用户体系（user, sys_user, sys_role, sys_menu 等）、课程体系（course, course_category, course_schedule, course_location 等）、订单体系（order, order_item, user_course_package）、教练体系（coach, coach_settlement）、AI 客服体系（ai_session, ai_message, ai_knowledge 等）、营销体系（banner, coupon 等）。核心流程是：用户下单 → 生成订单 → 支付后生成课包 → 上课销课 → 教练结算。"

### Q4：订单状态是怎么流转的？

> "订单有 7 个状态：待支付(1) → 支付后私教课进入处理中(3)，团课也进入处理中(3) → 完成销课后变为已完成(4)。用户可取消待支付订单变为已取消(5)。处理中的订单可申请退款进入退款中(6)，审核通过变为已退款(7)，驳回则回到处理中(3)。此外还有 15 分钟未支付自动取消的机制。"

### Q5：私教课和团课有什么区别？

| 维度 | 私教课 (type=1) | 团课 (type=2) |
|------|----------------|--------------|
| 教学模式 | 一对一 | 一对多 |
| 下单方式 | 直接购买课程 | 选择课程 + 具体排课场次 |
| 支付后 | 生成课包，按次消课 | 报名具体场次，按场次上课 |
| 有效期 | 有有效期（validity_days），过期作废 | 无课包概念，按场次上课 |
| 成团机制 | 无需成团 | 有最低成团人数（min_group_size） |
| 销课方式 | 教练手动消课（扣课包次数） | 团课统一签到 |
| 退款计算 | 按已消课节数折算 | 按时段（开课前 2h / 2-8h / 8h+）分档 |

### Q6：如何保证订单不会超卖？（团课名额控制）

> "使用了两个机制：1) 数据库层面，course_schedule 表有 enrolled_seats 和 total_seats 字段，报名时校验 enrolled_seats < total_seats；2) Redis 分布式锁，同一排课场次的报名操作串行化执行，防止并发超卖。分布式锁的 key 格式为 `order:user:schedule:create:{userId}:{scheduleId}` 和 `order:schedule:{scheduleId}`。"

### Q7：AI 客服是怎么实现的？

> "管理员在后台维护知识库，录入机构相关的常见问题和答案。学员在小程序 AI 客服页面发送消息后，后端将消息和知识库内容一起发送给大语言模型 API，模型基于知识库内容生成回答。如果 AI 无法解答，系统会自动标记为需要人工接管，由管理员在后台处理。"

### Q8：系统做了哪些安全措施？

> "1) 认证鉴权使用 Sa-Token，所有 API 都需要携带 token 访问；2) 密码使用 BCrypt 加密存储；3) 用户只能操作自己的订单（通过 userId 校验）；4) SQL 注入防护——MyBatis-Plus 使用参数化查询；5) 文件上传做了内容安全检测（ContentSecurityUtils）。"

### Q9：项目有哪些不足或可以改进的地方？

> "1) 目前推荐算法使用的是基于用户行为的简单协同过滤，可以引入更复杂的深度学习推荐模型；2) 支付模块目前是模拟支付，可以对接真实支付接口（微信支付）；3) 可以考虑引入 Elasticsearch 做课程全文搜索；4) 可以用 WebSocket 实现 AI 客服的流式对话体验。"

### Q10：你在项目中主要负责什么？遇到了哪些困难？

**建议回答思路**：

> "我负责了从需求分析、数据库设计到前后端开发的全部工作。最大的困难是理解 Spring Boot 的自动配置和依赖注入机制，因为之前只学过 Java SE 基础语法。通过阅读官方文档和调试代码，逐渐理解了 IoC 容器的原理。另一个难点是订单状态机的设计，需要考虑各种边界情况（超时取消、退款审核、成团判断等），通过梳理状态流转图最终完成了设计。"
