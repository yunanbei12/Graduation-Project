# 体育教培机构管理系统答辩速记版

## 1. 一句话介绍项目

这是一个面向体育教培机构的综合管理系统，包含学员端微信小程序、管理员端 Web 后台和 Java 后端服务，主要解决课程展示、报名下单、排课销课、教练结算以及智能客服与个性化推荐等业务问题。

### 2.2 3 分钟版本

各位老师好，我的毕业设计题目是体育教培机构管理系统。这个项目的目标是为体育培训机构提供一套相对完整的信息化管理方案，解决传统线下管理中课程安排混乱、订单跟踪不清晰、用户服务效率低和人工统计成本高的问题。

从系统结构上看，本项目采用前后端分离架构，共有三个端。第一个是学员端微信小程序，主要提供课程浏览、教练查看、课程报名、订单查询、课包查看、优惠券查看以及 AI 客服咨询等功能。第二个是管理员端 Web 后台，主要负责课程管理、排课管理、订单管理、销课管理、教练管理、营销管理、财务统计和 AI 知识库维护。第三个是后端服务，负责统一处理业务逻辑、权限认证、数据库访问以及消息通知等工作。

技术实现上，后端采用 Spring Boot 搭建，使用 MyBatis-Plus 简化数据库操作，使用 Sa-Token 处理登录认证和权限控制，使用 Redis 和 Redisson 解决团课报名时的并发问题，使用 RabbitMQ 实现订单超时自动取消和成团检查等异步任务。管理后台采用 Vue 3、Vite、Element Plus 构建，小程序端采用 uni-app 来实现多端兼容能力。

业务上，这个系统的一个特点是同时支持私教课和团课两种模式。私教课更偏向一对一教学，支付后会生成课包，后续通过销课来扣减课时；团课则需要选择具体场次报名，并且要考虑名额控制、最低成团人数和开课前退款规则等逻辑。除此之外，我还实现了一个 AI 智能客服模块，支持学员查询课程、订单、课包和优惠券信息，在无法解答时还能转人工处理。另一个亮点是个性化推荐模块，它会结合用户浏览、下单和签到行为，对课程进行排序推荐。

整体来说，这个项目比较完整地覆盖了体育教培机构从获客、报名、上课到售后的主要业务流程，也让我对前后端分离系统、数据库设计和业务流程建模有了更深入的理解。

## 5. 技术栈怎么说最稳

1. 后端使用 Spring Boot + Java 17，优点是生态成熟、开发效率高
2. 数据访问使用 MyBatis-Plus，优点是减少重复 CRUD 代码
3. 权限认证使用 Sa-Token，优点是相对轻量，适合本项目
4. 数据库使用 MySQL，适合存储课程、订单、用户等结构化数据
5. Redis + Redisson 用来处理并发报名和锁控制
6. RabbitMQ 用来实现订单超时取消和异步业务处理
7. 管理端使用 Vue 3 + Element Plus，适合快速搭建后台
8. 小程序端使用 uni-app，可以复用 Vue 风格语法

## 6. 业务流程怎么讲

### 6.1 私教课流程

用户选择私教课后下单，支付成功后系统会生成对应课包。学员上课后，管理员或教练在后台进行销课，系统会扣减课包中的剩余课时。课程完成后，相关数据会进入统计和结算流程。

### 6.2 团课流程

用户先选择团课，再选择具体排课场次进行报名。系统会校验该场次是否还有剩余名额，报名成功后生成订单。若报名人数达到最小成团人数，则课程正常开课；若未成团，则系统会自动取消或调整状态。

### 6.3 AI 客服流程

用户在小程序发送咨询消息后，后端会先识别用户意图，再结合知识库和业务数据生成回答。如果用户问题涉及订单、课包、优惠券等信息，系统会返回结构化结果；如果系统无法处理，则支持转人工。

## 8. 高频问答速记

### 8.1 你的项目解决了什么问题

主要解决体育培训机构在线下管理中课程安排不清晰、订单追踪不方便、销课统计麻烦以及用户咨询效率低的问题。

### 8.2 为什么做前后端分离

因为学员端和管理端面对的用户不同、页面和功能差异较大，前后端分离可以让前端专注页面交互，后端专注业务逻辑，也更方便后期扩展和维护。

### 8.3 为什么要分成两个后端服务

一个服务主要面向小程序用户端，一个服务主要面向管理员后台。这样接口职责更清晰，也方便分别做权限和业务隔离。

### 8.4 订单超时取消是怎么做的

用户创建订单后，系统会发送一个延迟消息到消息队列。如果 15 分钟内没有支付，消费者会自动执行取消订单逻辑，这样可以避免一直靠前端轮询或人工处理。

### 8.5 如何防止团课超卖

系统在用户报名团课时会先校验剩余名额，同时使用 Redis 分布式锁把同一用户和同一场次的关键操作串行化，避免高并发下重复报名或人数超限。

### 8.6 AI 客服的核心思路是什么

核心思路是“规则识别 + 知识库 + 业务数据 + 模型生成”。也就是说，不是单纯调用大模型，而是先判断问题类型，再把系统内部可查询的数据组织好，最后生成回答。

### 8.7 推荐系统怎么实现

推荐模块会参考用户的浏览、下单、签到等行为信息，同时结合课程类别、课程类型、价格区间、热度和可报名情况，对课程进行综合打分排序。

### 8.8 项目最大的难点是什么

最大的难点是订单状态流转和团课业务规则设计。因为不仅要考虑下单和支付，还要考虑自动取消、成团判断、退款审核、课包生成和销课扣次等边界情况。

### 8.9 你做了哪些安全控制

系统使用 token 做登录认证，后台菜单权限通过角色与菜单关系控制，密码采用加密存储，用户在查询订单等操作时只能访问自己的数据。

### 8.10 项目有哪些不足

1. 支付目前更偏演示流程，没有完整接入真实支付网关
2. 推荐算法还是规则加权，后续可以升级为更复杂的模型
3. AI 客服目前更适合常见问答，复杂场景仍需人工兜底
4. 还可以进一步增加日志监控、搜索能力和流式对话体验
