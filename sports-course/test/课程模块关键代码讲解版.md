## 3. 代码片段一：团课下单为什么要锁“用户+场次”

### 3.1 代码位置

1. [OrderServiceImpl.java](/Users/atrox/集美大学/Graduation-Project/sports-course/shop-back-end/kinetic-sports-service/src/main/java/com/kinetic/sports/service/impl/OrderServiceImpl.java:78)

### 3.2 建议展示片段

```java
if (Objects.equals(course.getType(), 2)) {
    // 团课订单必须绑定具体场次；这里先锁住“用户+场次”维度，避免重复下单。
    if (params.getScheduleId() == null) {
        throw new IllegalArgumentException("请选择上课时间");
    }
    userScheduleLock = redissonClient.getLock(
            "order:user:schedule:create:" + userId + ":" + params.getScheduleId()
    );
    userScheduleLock.lock();

    CourseSchedule schedule = courseScheduleService.getById(params.getScheduleId());
    validateScheduleForSignup(course, schedule);

    long existCount = this.count(new LambdaQueryWrapper<Order>()
            .eq(Order::getUserId, userId)
            .eq(Order::getScheduleId, params.getScheduleId())
            .in(Order::getStatus, ORDER_PENDING, ORDER_PAID, ORDER_PROCESSING, ORDER_FINISHED, ORDER_REFUNDING));
    if (existCount > 0) {
        throw new IllegalArgumentException("您已报名该场次，请勿重复购买");
    }
    order.setScheduleId(params.getScheduleId());
}
```

### 3.3 这一段在解决什么问题

这一段主要解决两个问题：

1. 团课必须绑定具体场次，不能像私教课一样直接买课包
2. 防止同一用户重复报名同一个团课场次

### 3.4 你可以这样讲

“这里是团课下单的核心分支。因为团课报名的是某一个具体排课场次，所以订单里必须带 `scheduleId`。我在这里先对‘用户+场次’做了 Redisson 分布式锁，避免用户连续点击造成重复下单。加锁以后，系统会继续校验这个场次是否存在、是否还能报名，以及当前用户是否已经报过同一场次。这样做的目的，是在业务层面防止重复报名，而不是只依赖前端按钮防抖。”

### 3.5 这一段体现的亮点

1. 团课和私教课业务模型不同
2. 使用 Redis 分布式锁做并发控制
3. 业务校验不是前端做样子，而是在后端真正落地

## 4. 代码片段二：为什么私教课支付后生成课包，而团课支付后占用名额

### 4.1 代码位置

1. [OrderServiceImpl.java](/Users/atrox/集美大学/Graduation-Project/sports-course/shop-back-end/kinetic-sports-service/src/main/java/com/kinetic/sports/service/impl/OrderServiceImpl.java:633)

### 4.2 建议展示片段

```java
if (Objects.equals(course.getType(), 1)) {
    // 私教课支付成功后不是占用场次，而是生成后续可销课的课包。
    order.setStatus(ORDER_PROCESSING);

    UserCoursePackage pkg = new UserCoursePackage();
    pkg.setUserId(order.getUserId());
    pkg.setCourseId(course.getId());
    pkg.setOrderId(order.getId());
    pkg.setTotalLessons(course.getLessonCount() != null ? course.getLessonCount() : 1);
    pkg.setUsedLessons(0);
    pkg.setExpireTime(LocalDateTime.now().plusDays(validityDays));
    pkg.setStatus(1);
    userCoursePackageService.save(pkg);

    course.setSales(defaultInt(course.getSales()) + 1);
    courseService.updateById(course);
    return;
}

// 团课支付成功后才真正占用名额，因此这里对场次加锁并递增已报名人数。
RLock scheduleLock = redissonClient.getLock("order:schedule:" + order.getScheduleId());
scheduleLock.lock();
try {
    CourseSchedule schedule = courseScheduleService.getById(order.getScheduleId());
    validateScheduleForSignup(course, schedule);
    schedule.setEnrolledSeats(defaultInt(schedule.getEnrolledSeats()) + 1);
    courseScheduleService.updateById(schedule);
} finally {
    scheduleLock.unlock();
}
```

### 4.3 这一段在解决什么问题

这一段体现了私教课和团课在支付后的核心差异：

1. 私教课支付成功后生成课包
2. 团课支付成功后占用场次名额

### 4.4 你可以这样讲

“这一段代码是我整个课程模块里最能体现业务差异的一段。私教课支付后，系统不会去绑定某个具体场次，而是生成一个用户课包，后续通过销课逐次扣减课时。团课正好相反，用户支付后，占用的是某个具体场次的报名名额，所以这里需要对场次再次加锁，并把 `enrolledSeats` 加一。也就是说，私教课的履约单位是课时，团课的履约单位是场次，这是两套不同的业务模型。”

### 4.5 这一段体现的亮点

1. 对真实业务有建模，而不是把所有课程都当成同一类商品
2. 统一订单模型下做课程类型分流
3. 支付成功后对不同业务走不同履约链路

## 5. 代码片段三：团课成团判断到底是怎么实现的

### 5.1 代码位置

1. [OrderServiceImpl.java](/Users/atrox/集美大学/Graduation-Project/sports-course/shop-back-end/kinetic-sports-service/src/main/java/com/kinetic/sports/service/impl/OrderServiceImpl.java:297)

### 5.2 建议展示片段

```java
Course course = courseService.getById(schedule.getCourseId());
int minGroupSize = course != null && course.getMinGroupSize() != null ? course.getMinGroupSize() : 1;
long enrolledCount = this.count(new LambdaQueryWrapper<Order>()
        .eq(Order::getScheduleId, scheduleId)
        .in(Order::getStatus, ORDER_PAID, ORDER_PROCESSING, ORDER_FINISHED));

LocalDateTime scheduleStart = resolveStartTime(schedule, course);
LocalDateTime checkTime = scheduleStart == null
        ? null
        : scheduleStart.minusMinutes(orderBizProperties.getGroupCheckBeforeMinutes());
boolean canFinalizeGroup = checkTime == null || !LocalDateTime.now().isBefore(checkTime);

if (enrolledCount >= minGroupSize) {
    // 达到成团人数后，场次进入可履约状态；当前配置下通常在开课时触发。
    if (Objects.equals(schedule.getStatus(), 0)) {
        schedule.setStatus(1);
        courseScheduleService.updateById(schedule);
    }
    return result;
}

if (!canFinalizeGroup) {
    return result;
}

if (!Objects.equals(schedule.getStatus(), 3)) {
    // 到了成团判断时间仍未成团，则取消场次并回滚相关订单。
    schedule.setStatus(3);
    courseScheduleService.updateById(schedule);
}
```

### 5.3 这一段在解决什么问题

这一段代码负责在开课时间判断：

1. 当前场次是否达到最少成团人数
2. 达到则正常开课
3. 未达到则取消场次

### 5.4 你可以这样讲

“这里是团课成团判断的核心实现。系统会先统计这个场次当前有多少有效报名订单，再读取课程设置的最少成团人数。然后通过 `checkTime` 判断当前是否已经到了成团判断时间。现在我的配置已经改成了开课时判断，所以如果开课时人数达到要求，场次就进入进行中；如果人数不够，系统就把场次改成已取消，并触发后面的订单取消和退款逻辑。这样就把现实中的‘成团规则’真正落到了代码里。”

### 5.5 这一段体现的亮点

1. 成团人数是动态业务规则，不是写死的常量
2. 成团判断不是人工判断，而是系统自动判断
3. 业务状态流转和时间规则结合在一起

List<Order> pendingOrders = this.list(new LambdaQueryWrapper<Order>()

### 6.3 这一段在解决什么问题

2. 已支付或进行中的订单自动退款

### 6.4 你可以这样讲

“老师如果问系统为什么会自动退款，我就展示这一段。这里不是简单只改排课状态，而是继续把关联订单同步处理掉。对于还没支付的订单，系统自动取消；对于已经支付的订单，系统自动退款。这样可以保证排课和订单状态一致，不会出现‘场次已经取消，但用户订单还显示正常’这种脏数据。”

### 6.5 这一段体现的亮点

## 7. 代码片段五：团课结课为什么不是简单改个状态

### 7.1 代码位置

1. [CourseCheckinServiceImpl.java](/Users/atrox/集美大学/Graduation-Project/sports-course/shop-back-end/kinetic-sports-service/src/main/java/com/kinetic/sports/service/impl/CourseCheckinServiceImpl.java:46)

### 7.2 建议展示片段

```java
LocalDateTime scheduleEnd = resolveEndTime(schedule, course);
// 团课必须在真正下课后才能结课，避免后台提前生成签到和完成订单。
if (scheduleEnd != null && LocalDateTime.now().isBefore(scheduleEnd)) {
    throw new IllegalArgumentException("课程未结束，暂不能结课");
}

List<Order> paidOrders = orderMapper.selectList(new LambdaQueryWrapper<Order>()
        .eq(Order::getScheduleId, scheduleId)
        .in(Order::getStatus, 2, 3, 4));

for (Order order : paidOrders) {
    // 团课结课的本质是：按场次为每个报名学员生成签到，并同步收口订单状态。
    CourseCheckin checkin = new CourseCheckin();
    checkin.setUserId(order.getUserId());
    checkin.setScheduleId(scheduleId);
    checkin.setCheckinType(2);
    checkin.setStatus(isAbsent ? 0 : 1);
    checkins.add(checkin);

    if (!Objects.equals(order.getStatus(), 4)) {
        order.setStatus(4);
        order.setFinishTime(finishTime);
        orderMapper.updateById(order);
    }
}

// 只有签到和订单都处理完成后，才把排课状态收口为已结课。
schedule.setStatus(2);
courseScheduleMapper.updateById(schedule);
```

### 7.3 这一段在解决什么问题

这一段说明团课结课不是一个展示动作，而是一次业务收口：

1. 必须先确认课程真的结束
2. 为每个报名学员生成签到记录
3. 把订单状态更新为已完成
4. 最后才更新排课状态

### 7.4 你可以这样讲

“这里是团课结课的核心逻辑。结课不是只把排课状态改成已结课，而是要先判断下课时间是否已到，然后按场次为每个报名学员生成签到记录，再把对应订单状态同步改成已完成，最后才把排课收口。也就是说，这一步同时影响签到表、订单表和排课表，所以必须放在同一套业务逻辑里完成。”

### 7.5 这一段体现的亮点

1. 结课逻辑完整，不是伪功能
2. 多表一致性处理清楚
3. 团课签到和订单完成真正打通

## 8. 代码片段六：为什么还要加定时任务兜底

### 8.1 代码位置

### 8.2 建议展示片段

```java
LocalDateTime scheduleStart = resolveStartTime(schedule, course);
// 兜底任务只处理“已经到开课时间”的场次，避免提前干预正常报名流程。
if (scheduleStart == null || now.isBefore(scheduleStart)) {
    continue;

if (enrolledCount < minGroupSize) {

    orderService.checkGroupSchedule(schedule.getId());

    schedule.setStatus(1);

```

以及：

```java
if (scheduleEnd != null && now.isAfter(scheduleEnd)) {
    // 管理员忘记手动结课时，定时任务会补做结课，避免订单和排课长期停在中间态。
    if (!courseCheckinService.hasGroupCheckins(schedule.getId())) {
        courseCheckinService.settleGroupSchedule(schedule.getId(), List.of());
    } else {
        schedule.setStatus(2);
        courseScheduleService.updateById(schedule);
    }
}
```

### 8.3 这一段在解决什么问题

定时任务解决的是“人工操作不一定稳定”的问题：

1. 到了开课时间但 MQ 没有触发或者管理员没管，系统仍能补做成团判断
2. 到了下课时间但管理员忘了手动结课，系统仍能补做结课

### 8.4 你可以这样讲

“这里我额外加了一个定时任务做兜底。原因是，真实业务里不能假设每一次异步消息、每一次人工操作都刚好执行成功。所以我让系统每隔一段时间扫描一次已经到开课时间但还没开始的场次，自动补做成团判断；对于已经过下课时间但还没结课的场次，自动补做结课。这样可以避免订单、排课和签到长期停留在中间状态。”

### 8.5 这一段体现的亮点

1. 你考虑到了异常与补偿机制
2. 不是只做主流程，还做了系统稳定性设计
3. 体现出后端业务系统思维

## 10. 90 秒代码讲解示范稿

“这里我展示的是课程订单支付后的核心处理逻辑。这个逻辑先根据课程类型做分流。如果是私教课，支付成功后系统会生成一个课包，因为私教课后续是通过销课逐次扣减课时的；如果是团课，支付成功后系统占用的是某个具体排课场次的名额，所以这里会对场次加锁并递增已报名人数。

后面在开课时间，系统会执行成团判断。如果当前报名人数达到最少成团人数，场次正常进入进行中；如果人数不足，系统就自动取消场次，并对相关订单做取消或退款处理。课程结束后，后台结课逻辑会为每个报名学员生成签到记录，并把订单状态更新为已完成。也就是说，这套代码串起来解决的是从支付、履约到收口的完整业务流程，而不是只做一个简单下单功能。”
