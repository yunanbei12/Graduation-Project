# 课程模块完整链路与答辩演示文档

## 1. 文档用途

这份文档用于说明本项目中“私教课”和“团课”两类课程的完整业务链路、核心实现逻辑，以及毕业答辩时可直接使用的演示顺序和讲解话术。

它的定位不是源码注释，而是站在“业务闭环 + 技术实现 + 答辩表达”的角度，把课程模块从前台到后台、从下单到上课、从签到到结算的整体逻辑讲清楚。

---

## 2. 课程模块整体设计思路

本系统中的课程业务不是单一模型，而是按业务特点拆分为两类：

1. 私教课
2. 团课

两类课程在“浏览、下单、支付、履约、退款、结算”这些环节上有共性，但在实际业务规则上差异很大：

1. 私教课更像“购买课包”，支付后生成课包，后续通过销课逐次扣减课时。
2. 团课更像“报名具体场次”，用户必须选择某个排课场次报名，并受到座位数、成团人数、开课时间和结课签到等规则约束。

因此，系统采用了“统一订单模型 + 区分课程类型处理逻辑”的方式：

1. 订单表统一承载课程订单。
2. 私教课支付后生成 `user_course_package`。
3. 团课支付后绑定 `scheduleId`，并走成团判断、排课履约和团课签到逻辑。

这样设计的好处是：

1. 数据模型统一，便于订单管理和财务统计。
2. 业务规则分开，避免私教课和团课强行共用一套流程导致逻辑混乱。
3. 答辩时更容易体现“系统理解了真实业务差异”，而不是简单做了两个页面。

## 3. 系统结构与课程模块位置

### 3.1 前端端口划分

课程业务涉及两个前端：

1. 学员端小程序 `sports-custom-mini`
2. 管理后台 `sports-admin-web`

### 3.2 后端模块划分

课程业务相关的主要后端模块有：

1. `kinetic-sports-api`：给小程序提供接口
2. `kinetic-sports-admin`：给管理后台提供接口
3. `kinetic-sports-service`：核心业务逻辑层
4. `kinetic-sports-bean`：实体类、DTO、VO
5. `kinetic-sports-common`：公共配置、统一响应、异常处理

### 3.3 课程业务核心数据对象

课程模块中最关键的几个表或模型如下：

1. `course`：课程基础信息，区分私教课和团课
2. `course_schedule`：团课具体排课场次
3. `order`：课程订单
4. `order_item`：订单项
5. `user_course_package`：私教课包
6. `course_checkin`：签到与销课记录
7. `coach_settlement`：教练结算记录
8. `coupon`、`user_coupon`：优惠券及用户优惠券

其中最关键的字段有：

1. `course.type`：`1=私教课`，`2=团课`
2. `order.scheduleId`：团课订单关联的排课场次
3. `course_schedule.status`：`0=未开始`，`1=进行中`，`2=已结课`，`3=已取消`
4. `course_checkin.checkinType`：`1=私教销课`，`2=团课签到`

## 4. 私教课完整业务链路

### 4.1 用户端业务流程

私教课在学员端的典型流程如下：

1. 用户进入课程列表页，筛选私教课。
2. 点击进入私教课程详情页。
3. 在确认订单页直接提交订单。
4. 完成支付后，订单进入履约阶段。
5. 后台在实际上课后执行销课。
6. 系统扣减课包剩余节数。

### 4.3 私教课支付后为什么生成课包

私教课与团课最大的不同在于：私教课支付的不是某一次固定场次，而是一组后续可消耗的课时，因此支付成功后系统会自动生成课包。

1. 订单状态改为 `进行中(3)`
2. 创建 `UserCoursePackage`
3. 保存总课时、已用课时、有效期等信息
4. 更新课程销量

### 4.4 私教销课与订单完成

私教课不是在支付时就完成，而是在后台每次上课后执行销课。

后台销课接口：

1. `/checkin/private`

销课过程包括：

1. 校验学员、课程、教练和上课时间
2. 查找该学员当前最早到期且仍可用的课包
3. 生成一条 `course_checkin` 记录
4. 课包已用节数 `usedLessons + 1`
5. 当课包全部使用完时，将课包状态改为已完成
6. 如果课包已完成，则把对应订单状态更新为 `已完成(4)`

### 4.5 私教课退款逻辑

私教课退款不是简单全额退款，而是根据已消耗课时折算退款金额。

系统支持先预览退款金额，再提交退款申请：

1. `/order/refund-preview/{id}`
2. `/order/refund/{id}`

退款计算规则：

1. 单节价格 = 课程原价 / 总课时数
2. 已消耗金额 = 单节价格 × 已销课节数
3. 可退金额 = 实付金额 - 已消耗金额
4. 退款后课包失效

## 5. 团课完整业务链路

### 5.1 用户端业务流程

团课的典型流程如下：

1. 用户进入课程列表页，选择团课。
2. 进入团课详情页查看可报名场次。
3. 选择一个或多个未来场次。
4. 进入团课确认订单页。
5. 提交订单并支付。
6. 系统记录报名人数，并在开课时间执行成团判断。
7. 成团则正常开课，不成团则自动取消并退款。
8. 课程结束后由后台执行结课，生成签到记录。
9. 团课签到数据进入后续教练结算流程。

小程序相关页面主要包括：

1. `pages/course/course-detail-group.vue`
2. `pages/order/confirm-group.vue`
3. `pages/order/order-detail-course.vue`

### 5.2 团课为什么必须绑定具体场次

团课不是先买课包再约课，而是直接报名某个具体时间段，因此订单必须绑定排课场次。

系统通过 `order.scheduleId` 把团课订单和 `course_schedule` 关联起来，这样后续才能处理：

1. 名额控制
2. 重复报名校验
3. 成团判断
4. 开课前退款规则
5. 结课签到

这也是团课与私教课在业务建模上的根本差异。

### 5.3 团课排课管理

后台管理员先创建团课模板，再按日期生成具体排课。

排课管理支持：

1. 单条新增排课
2. 批量新增排课
3. 编辑排课
4. 教练冲突校验
5. 总座位数与成团人数校验

4. 校验场次与课程是否匹配
5. 校验排课状态是否允许报名
6. 校验当前时间是否还未开课
7. 校验剩余名额是否充足
8. 校验用户是否已经报名同一场次

3. `enrolledSeats + 1`

## 6. 课程订单状态流转说明

本系统课程订单主要涉及以下状态：

1. `1=待支付`
2. `2=已支付`
3. `3=进行中`
4. `4=已完成`
5. `5=已取消`
6. `6=退款中`
7. `7=已退款`

### 6.1 私教课状态流转

私教课一般是：

1. 创建订单 -> `待支付`
2. 支付成功 -> `进行中`
3. 课包全部销完 -> `已完成`
4. 中途退款 -> `退款中 / 已退款`

### 6.2 团课状态流转

团课一般是：

1. 创建订单 -> `待支付`
2. 支付成功 -> `进行中`
3. 开课时成团判断失败 -> `已退款`
4. 成团并完成上课 -> `已完成`
5. 用户中途申请退款 -> `退款中 / 已退款`

答辩时可以强调：订单状态不是静态标签，而是反映业务执行阶段的核心字段。

## 7. 优惠券、消息队列和锁在课程模块中的作用

### 7.1 优惠券

课程订单支持优惠券，主要流程是：

1. 下单前计算优惠金额
2. 创建订单时先锁定优惠券
3. 支付成功后把优惠券改为已使用
4. 订单取消或退款时按规则释放或回滚优惠券状态

### 7.2 RabbitMQ

消息队列在课程模块中主要负责两类异步任务：

1. 延迟关单
2. 团课成团检查

它的好处是：

1. 避免用户端轮询
2. 让超时关单和成团判断在后台自动执行
3. 提高业务处理的异步能力

### 7.3 Redis 与 Redisson

Redis 和 Redisson 主要用在团课报名和支付流程中：

1. 对同一用户、同一场次的报名创建加锁
2. 对同一场次的支付更新加锁
3. 防止重复报名和超卖

这部分是课程模块最有代表性的技术亮点之一。

## 8. 课程链路与财务结算的关系

课程模块并不是在“报名成功”后就结束，它还会继续流向财务结算。

### 8.1 私教课结算来源

私教课每次销课都会生成一条 `course_checkin` 记录，其中包含：

1. 单节价格快照
2. 教练分成比例
3. 教练应结算金额

### 8.2 团课结算来源

团课结课时同样会生成 `course_checkin` 记录，只不过不扣课包，而是记录该学员对该场次是否出勤。

### 8.3 教练结算逻辑

后台财务模块会：

## 10. 答辩演示逐段话术

### 10.1 开场介绍

各位老师好，我的毕业设计项目是体育教培机构管理系统。这个系统面向体育培训机构的日常运营，分为学员端小程序、管理员后台和后端服务三部分。系统的核心业务包括课程管理、团课排课、订单处理、销课签到、教练结算，以及 AI 智能客服和课程推荐等功能。

今天我重点演示课程模块，特别是团课从咨询、报名、支付、成团、上课、结课到结算的完整业务闭环。同时我也会补充说明私教课与团课在业务模型上的差异。

### 10.2 介绍私教课与团课的区别

在这个系统里，我把课程分成两种模式。第一种是私教课，用户支付后会生成课包，后续通过销课逐步扣减课时。第二种是团课，用户必须选择具体排课场次报名，并且要考虑座位数、最少成团人数、开课时间和结课签到等业务规则。

这样设计的原因是，虽然两者都属于课程订单，但真实业务逻辑差异很大。如果简单共用一套模型，会导致后续订单状态、退款规则和履约流程都变得混乱。

### 10.3 演示小程序端课程报名

现在我先打开小程序端。用户可以通过课程页浏览私教课和团课，也可以通过 AI 智能客服直接询问合适的课程或最近还有名额的团课场次。

这里我选择一个团课案例。进入课程详情页后，用户可以看到课程名称、价格、场次时间和已报名人数。团课不能像私教课一样直接购买课包，而是必须选择具体场次进行报名。

### 10.4 演示下单与支付

用户选择场次后进入确认订单页。这里系统会先展示课程费用、优惠券和实付金额。提交订单时，后端会校验用户是否已登录、是否已绑定手机号、课程是否存在、场次是否可报名、是否还有名额，以及用户是否重复报名同一场次。

为了防止高并发场景下出现超卖，我在后端对同一用户和同一场次的关键报名操作加了 Redis 分布式锁。只有当订单、订单项和优惠券锁定等步骤全部成功时，事务才会提交。

### 10.5 演示团课成团逻辑

团课支付成功后，系统会把这个用户记入已报名人数，并在开课时间执行成团判断。现在系统的逻辑是“开课时判断”，如果人数达到了课程设置的最少成团人数，排课就正常进入进行中状态；如果人数不足，系统会自动取消场次，并对已支付订单执行自动退款。

这部分逻辑体现了系统对真实业务规则的建模，而不是简单地把订单状态从待支付改成已支付就结束。

### 10.6 演示后台排课与结课

下面我切换到管理员后台。后台管理员可以查看课程、排课和订单信息。对于团课来说，管理员不仅要管理课程，还要管理每一个具体场次。

当课程结束后，管理员可以执行结课操作。系统会根据出勤情况生成团课签到记录，同时把相关订单状态更新为已完成，再把排课状态更新为已结课。这样就实现了从下单到上课再到课程收口的完整闭环。

### 10.7 演示私教课补充说明

如果是私教课，逻辑会有所不同。私教课支付成功后，系统会生成课包。后续每次上课时，后台执行销课，课包剩余节数会减少。只有当课包里的课时全部使用完，对应订单才会完成。

也就是说，私教课的履约核心是“课包扣减”，团课的履约核心是“场次签到”。

### 10.8 演示教练结算

结课完成后，系统中的签到记录还会继续流入财务结算模块。后台可以按教练和周期汇总未结算的签到记录，自动统计课时数和应结算金额，并生成教练结算单。

这样整个课程模块最终和财务模块也打通了，形成了完整的业务闭环。

### 10.9 结束总结

通过刚才的演示可以看到，这个系统中的课程模块并不是简单的课程展示和下单页面，而是覆盖了学员端报名、后台排课、订单状态流转、私教销课、团课成团、结课签到和教练结算等完整过程。

其中比较有代表性的技术点包括：使用 Redis 分布式锁防止团课超卖，使用 RabbitMQ 处理订单超时取消和团课成团检查，以及通过统一订单模型加区分课程类型逻辑来支持私教课和团课两种不同业务模式。

## 11. 老师可能追问的问题与回答

### 11.1 私教课和团课的核心区别是什么

私教课的核心是“支付后生成课包，后续逐次销课”，团课的核心是“支付后占用某个具体场次名额，并在开课时判断是否成团”。因此两者在履约、退款和完成条件上都不同。

### 11.2 为什么团课不直接支付后就算完成

因为团课支付成功只代表报名成功，不代表课程已经实际发生。它后面还需要经过成团判断、实际上课和结课签到，所以订单要等到课程结束后才能真正完成。

### 11.3 你是怎么防止团课超卖的

我在后端对同一用户、同一场次的报名和支付关键路径加了 Redisson 分布式锁，同时在支付时再次校验场次是否还有剩余名额，避免高并发情况下出现重复报名和人数超限。

### 11.4 为什么把成团判断改成开课时

因为如果提前太久判断，会出现“课程还允许报名，但用户刚支付就被自动退款”的体验问题。改成开课时判断后，业务更符合用户直觉，也更容易解释。

### 11.5 私教课退款为什么不是全部原价退

因为私教课的履约单位是课时。如果课包已经消耗了一部分节数，就不能再整单全额退款，所以系统会按照已消耗节数折算退款金额。

### 11.6 团课为什么需要排课表

因为团课天然依赖具体时间和地点。用户报名的不是抽象课程，而是某一个具体场次，所以必须有独立排课表来承载日期、时间段、教练、地点、座位数和状态等信息。

### 11.7 为什么说这个模块不是简单的增删改查

因为它除了基础数据管理外，还涉及订单状态流转、并发控制、延迟任务、自动退款、履约签到和财务结算等一系列真实业务规则。
