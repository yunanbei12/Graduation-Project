# 体育教培机构管理系统 - 需求分析文档

> 版本：v1.0  
> 日期：2026-03-21  

---

## 一、系统概述

### 1.1 系统定位
一个面向体育教培机构的 SaaS 管理系统，支持**私教上门**和**团课**两种业务模式，融合**拼团营销**、**智能推荐**、**AI 客服**、**体育电商**四大特色功能。

### 1.2 核心特色
1. **上门体育模式**：支持教练上门教学，地点由学员决定
2. **双模式拼团**：私教课包拼团（价格优惠）+ 团课拼团（成团上课）
3. **智能推荐**：基于规则混合推荐算法
4. **AI 客服**：接入大模型 API 的智能问答系统
5. **体育电商**：运动装备销售 + 课程装备套餐联动

### 1.3 技术栈
- 前端：`Vue3 + 微信小程序
- 后端：SpringBoot3
- 数据库：MySQL
- AI 服务：大模型 API（文心一言/通义千问）

**业务场景 1：私教课包拼团**

```
场景：张三发起"10 节私教课"拼团，原价 2000 元，拼团价 1500 元

流程：
1. 张三支付 1500 元发起拼团（选择课程包：10 节课）

4. 李四、王五看到后参与拼团并支付
5. 48 小时内满 3 人 → 成团 → 三人各自获得 10 节课时
6. 48 小时内未满 3 人 → 拼团失败 → 自动原路退款
```

**业务场景 2：团课拼团**

```
场景：后台创建"周六下午幸福小区羽毛球团课"

1. 后台创建团课：时间（周六 15:00）、地点（幸福小区羽毛球场）、教练、需 5 人成团、单价 50 元

3. 用户支付报名（可先收款，不成团再退）

   - 成功：微信订阅消息通知上课时间地点
   - 失败：自动取消课程 + 自动原路退款
5. 后台也可手动取消团课（触发自动退款）
```

#### 3.1.1.1 私教课包退款规则

**退款公式**：
```
退款金额 = (实付金额 ÷ 总课时数) × 剩余课时数
```

**计算示例**：
```
场景：用户购买私教课包
- 实付金额：1200 元
- 总课时：12 节
- 已上课时：2 节
- 剩余课时：10 节

计算：
- 每节课时单价：1200 ÷ 12 = 100 元/节
- 退款金额：100 × 10 = 1000 元
```

**退款流程**：
```
1. 用户小程序申请退款（选择课包、填写原因）
2. 系统自动计算退款金额
3. 管理员后台审核退款申请
4. 审核通过 → 调用支付接口原路退款
5. 退款成功 → 微信通知用户
6. 更新课时状态（标记为已退款）
```

**退款规则**：
- 随时可申请退款（无时间限制）
- 按剩余课时等比例退款（公平合理）
- 原路退回（微信支付账户）
- 退款后课时作废（不可恢复）

**拼团课包退款特殊规则**：
```
条款：拼团成功后的课包在有效期内若产生退款，需按原价重新核算已消耗课时

原因：
- 拼团价是优惠价（如 1500 元），原价更高（如 2000 元）
- 如果按拼团价退款，用户已上课时也按拼团价计算
- 会出现：用户上了大部分课，最后退款还能赚差价的漏洞

计算逻辑：
1. 用户购买拼团课包：原价 2000 元，拼团价 1500 元，12 节课
2. 用户上了 6 节课后申请退款
3. 系统按原价重新核算：
   - 原价单价：2000 ÷ 12 = 166.67 元/节
   - 已消耗课时价值：166.67 × 6 = 1000 元
   - 退款金额：实付 1500 - 已消耗 1000 = 500 元
4. 如果已消耗课时价值 ≥ 实付金额，则不予退款

示例场景：
场景 A：上了 2 节课后退款
  - 已消耗：166.67 × 2 = 333.34 元
  - 退款：1500 - 333.34 = 1166.66 元

场景 B：上了 8 节课后退款
  - 已消耗：166.67 × 8 = 1333.36 元
  - 退款：1500 - 1333.36 = 166.64 元

场景 C：上了 10 节课后退款
  - 已消耗：166.67 × 10 = 1666.7 元
  - 已消耗 > 实付金额，不予退款

防钻空子机制：
- 拼团课包退款时，自动触发原价核算
- 系统自动计算，无需人工干预
- 明确告知用户退款规则（小程序端展示）
- 防止：低价拼团 → 上大部分课 → 高价退款的套利行为
```

**功能清单**：
- [ ] 用户申请退款（小程序端）
- [ ] 系统自动计算退款金额
- [ ] 拼团课包自动按原价核算

### 3.5 智能推荐模块（毕设亮点）

#### 3.5.1 推荐算法设计

**混合推荐公式**：

**冷启动策略（三阶段演进）**：

```
【阶段 1：新用户（兴趣选座）】
场景：用户首次登录小程序，User Profile 为空
策略：主动收集用户偏好

流程：
1. 弹出“兴趣选座”引导页（可选跳过）
   ┌──────────────────────┐
   │  欢迎加入！           │
   │  告诉我们你的兴趣     │
   │                      │
   │  喜欢的运动（多选）： │
   │  ☑ 篮球  ☐ 羽毛球   │
   │  ☐ 足球  ☐ 网球     │
   │                      │
   │  期待地点：           │
   │  ☑ 幸福小区          │
   │  ☐ 阳光花园          │
   │                      │
   │  课程偏好：           │
   │  ○ 私教课            │
   │  ○ 团课              │
   │  ○ 都可以            │
   │                      │
   │  价格区间：           │
   │  ○ 500-1000 元       │
   │  ○ 1000-2000 元      │
   │  ○ 2000 元以上       │
   │                      │
   │  [跳过]  [开始探索]   │
   └──────────────────────┘

2. 用户选择后 → 生成初始 User Profile
   user_profile = {
     'sport_preference': ['篮球'],
     'location': '幸福小区',
     'preferred_type': '都可以',
     'price_range': (1000, 2000)
   }

3. 基于初始 User Profile 推荐课程
   personalization_score = 
     sport_match * 0.4 +  // 运动项目匹配（权重最高）
     location_match * 0.3 +  // 地点匹配
     type_match * 0.2 +  // 课程类型匹配
     price_match * 0.1  // 价格区间匹配

4. 如果用户跳过兴趣选座：
   → 按热度推荐（销量最高 + 运营权重）

【阶段 2：首次购买后】
触发条件：用户完成第 1 次购买
策略：兴趣选座 + 实际行为

更新 User Profile：
user_profile = {
  'sport_preference': ['篮球', '羽毛球'],  // 兴趣选座 + 实际购买
  'location': '幸福小区',
  'preferred_type': '私教课',  // 从实际购买中学习
  'price_range': (1000, 2000),
  'purchase_history': ['篮球私教课']  // 新增
}

推荐权重调整：
personalization_score = 
  sport_match * 0.35 +  // 降低兴趣选座权重
  purchase_match * 0.35 +  // 新增：实际购买匹配
  location_match * 0.15 +
  type_match * 0.1 +
  price_match * 0.05

【阶段 3：3 次以上购买】
触发条件：用户完成 3 次以上购买
策略：完整混合推荐算法

推荐权重：
personalization_score = 
  purchase_match * 0.4 +  // 实际行为为主
  sport_match * 0.2 +  // 兴趣选座为辅
  location_match * 0.2 +
  type_match * 0.1 +
  price_match * 0.1

完整公式：
recommend_score = 
  popularity_score * 0.4 +
  personalization_score * 0.4 +
  operation_weight * 0.2
```

**学术价值**（论文讨论点）：
```
1. 主动收集 vs 被动积累
   - 兴趣选座（主动）：用户明确表达偏好
   - 行为学习（被动）：从实际购买中推断
   - 对比实验：哪种方式冷启动效果更好？

2. 兴趣选座准确率评估
   - 用户选择的运动 vs 实际购买的课程
   - 准确率 = 匹配数 / 总选择数
   - 假设：准确率 70%，说明用户偏好会变化

3. 转化率对比实验
   - 填写兴趣选座用户：转化率 X%
   - 未填写兴趣选座用户：转化率 Y%
   - 预期：X > Y（精准推荐提高转化）

4. 推荐效果演进
   - 冷启动阶段（兴趣选座）：CTR（点击率） 5%
   - 首次购买后（混合）：CTR 8%
   - 成熟阶段（完整算法）：CTR 12%
   - 结论：数据越多，推荐越精准

5. 权重动态调整
   - 兴趣选座权重：40% → 35% → 20%（逐渐降低）
   - 实际行为权重：0% → 35% → 40%（逐渐升高）
   - 说明：实际行为比主动选择更可靠
```

**1. 热度分计算**：

#### 3.8.3.1 套餐拆分退款规则

**业务场景**：
```
场景：用户购买“新手篮球套餐”（10 节课 + 篮球），套餐价 1800 元
问题：用户收到篮球后，申请整个套餐退款怎么办？

风险：
- 用户收到篮球并使用
- 申请套餐全额退款
- 用户免费获得篮球，机构亏损
```

**退款规则**：
```
原则：实物商品一旦发货且非质量问题，退款时需扣除实物原价

情况 1：实物未发货
  → 全额退款（课程 + 实物）
  → 取消实物订单

情况 2：实物已发货，质量问题
  → 全额退款（课程 + 实物）
  → 实物可退回或保留（机构决定）

情况 3：实物已发货，非质量问题（最常见）
  → 拆分退款：
    * 实物部分：按原价扣除（不退款）
    * 课程部分：按剩余课时等比例退款
  → 实物不退回（用户已使用）
```

**计算示例**：
```
套餐信息：
- 套餐价：1800 元（优惠 400 元）
- 课程原价：2000 元/10 节（200 元/节）
- 实物原价：篮球 200 元

场景 A：实物未发货，申请退款
  - 退款金额：1800 元（全额）
  - 处理：取消课程 + 取消实物订单

场景 B：实物已发货，上了 3 节课后退款（非质量问题）
  - 课程原价单价：2000 ÷ 10 = 200 元/节
  - 已消耗课程价值：200 × 3 = 600 元
  - 实物扣除：200 元（原价）
  - 退款金额：1800 - 600 - 200 = 1000 元

场景 C：实物已发货，上了 8 节课后退款（非质量问题）
  - 已消耗课程价值：200 × 8 = 1600 元
  - 实物扣除：200 元（原价）
  - 退款金额：1800 - 1600 - 200 = 0 元
  - 已消耗 + 实物 ≥ 套餐价，不予退款

场景 D：实物已发货，质量问题退款
  - 退款金额：1800 元（全额）
  - 实物可退回或保留（机构决定）
```

**防钻空子机制**：
- 套餐退款时，自动判断实物发货状态
- 已发货且非质量问题 → 强制扣除实物原价
- 系统自动计算，无需人工干预
- 明确告知用户套餐退款规则（小程序端展示）
- 防止：购买套餐 → 使用实物 → 申请全额退款的套利行为

```
套餐内容：10 节私教课 + 篮球 + 运动鞋
  ↓
套餐价：1800 元（原价 2200 元）
  ↓
微信支付
  ↓
支付成功
  ↓
系统拆分订单：
  - 课程部分：10 节课时到账
  - 实物部分：生成商品订单
  ↓
课程部分：学员可在小程序查看课时
  ↓
实物部分：后台发货 → 用户确认收货
```

### 4.8 套餐拆分退款流程
```
用户申请“新手篮球套餐”退款
  ↓
[判断] 实物商品是否已发货？
  ├─ 未发货 → 全额退款（课程 + 实物）
  └─ 已发货 → 进入下一步判断
  ↓
[判断] 实物商品是否质量问题？
  ├─ 是 → 全额退款（课程 + 实物，实物可退可退）
  └─ 否 → 进入拆分退款逻辑
  ↓
拆分退款计算：
  - 实物部分：按原价扣除（不退款）
  - 课程部分：按剩余课时等比例退款
  ↓
示例计算：
  - 套餐价：1800 元（优惠 400 元）
  - 课程原价：2000 元/10 节
  - 实物原价：篮球 200 元
  - 用户已上课：3 节，剩余 7 节
  - 实物已发货，非质量问题
  ↓
退款金额计算：
  1. 课程原价单价：2000 ÷ 10 = 200 元/节
  2. 已消耗课程价值：200 × 3 = 600 元
  3. 实物扣除：200 元（原价）
  4. 退款金额：1800 - 600 - 200 = 1000 元
  ↓
退款完成 → 微信通知用户
```
