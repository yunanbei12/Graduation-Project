## 一、文档目标

本文档不是答辩展示文档，而是后续开发执行文档，核心目标如下：

1. 将现有需求、架构、数据库、接口设计落成可执行的开发计划
2. 按“后端、小程序端、后台管理端”三套代码工程拆分任务
3. 明确每个阶段的开发重点、依赖关系、交付物和验收标准
4. 降低后续开发过程中重复分析、遗漏模块、顺序错误的问题

## 二、开发边界与代码组织约定

### 2.1 三个代码工程约定

后续代码统一只放 3 个顶层工程中，不再拆第四个业务工程：

1. `backend/`
   用于放置后端代码，技术栈为 `Spring Boot 3 + Java 17 + MyBatis-Plus`
2. `miniapp/`
   用于放置用户端代码，技术栈为 `uni-app + Vue3 + TypeScript`
3. `admin-web/`
   用于放置后台管理端代码，技术栈为 `Vue3 + Vite + TypeScript + Element Plus`

当前仓库中的 `文档/`、`原型/` 继续保留，不放业务运行代码。

### 2.2 后续目录建议

建议未来代码目录结构如下：

```text
Graduation-Project
├── backend/          # 后端服务
├── miniapp/          # 小程序端
├── admin-web/        # 后台管理端
├── 文档/              # 设计文档、SQL、OpenAPI
└── 原型/              # 现有原型
```

### 2.3 三端职责边界

### 2.3.1 `backend/`

负责：

1. 业务规则实现
2. 数据库交互
3. 接口提供
4. 支付、退款、拼团、课时、推荐、AI、消息等核心流程

不负责：

1. 页面展示
2. 样式实现
3. 小程序端交互状态管理

### 2.3.2 `miniapp/`

负责：

1. 学员/家长端页面实现
2. 课程、拼团、订单、商品、动态、AI 客服、成长档案等用户交互
3. 支付调起、页面状态展示

不负责：

1. 核心金额计算
2. 拼团成团判断
3. 退款金额计算

### 2.3.3 `admin-web/`

负责：

1. 后台管理页面
2. 课程、学员、教练、排课、订单、退款、商品、动态、FAQ、财务报表等管理功能
3. 后台操作入口与数据展示

不负责：

1. 用户端分享、支付调起
2. 后端领域规则计算

## 三、开发总策略

### 3.1 开发顺序原则

为了保证闭环最早可用，开发顺序采用：

1. 先打基础环境，再做后端核心领域
2. 先做后台管理最小闭环，再做小程序购买闭环
3. 先完成 P0 闭环，再补 P1 亮点功能
4. 所有前端页面以现有原型为视觉参考，以 API 文档为接口依据

### 3.2 推荐执行方式

推荐采用“单人主导 + AI 协作”的节奏，按周迭代推进。

每周固定输出：

1. 本周完成的后端接口与数据库落地情况
2. 本周完成的小程序页面和后台页面
3. 本周联调结果与问题清单
4. 下周任务调整项

### 3.3 开发优先级执行规则

严格按以下优先级推进：

1. P0 必须先完成
2. P1 在 P0 形成闭环后逐步补齐
3. P2 只在时间充足时进行

## 四、开发周期总览

### 4.1 周期假设

本开发周期按单人开发估算，建议周期为 **12 周**。

其中：

1. 第 1-6 周：P0 核心闭环
2. 第 7-10 周：P1 毕设亮点
3. 第 11-12 周：联调优化、部署、演示与补文档

| 阶段 | 周次 | 核心目标 |

## 七、模块依赖关系

### 7.1 强依赖关系

以下模块必须按顺序推进：

1. 用户认证 -> 课程/商品浏览 -> 订单 -> 支付
2. 支付 -> 课时账户发放 -> 销课记录
3. 拼团 -> 支付 -> 成团判断 -> 退款
4. 商品订单 -> 发货 -> 确认收货
